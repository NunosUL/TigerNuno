Scoring & Confidence

How individual search results are scored using hybrid RRF fusion, and how those scores — combined with chunk count and LLM language — determine the confidence signal shown on every answer.

Part 1 — How search results are scored, reranked & filtered

Hybrid Search Scoring — RRF Fusion

⚡ Vector Similarity Search

The user's question is embedded into the same 3072-dimensional vector space used during ingestion (text-embedding-3-large). Azure AI Search computes the cosine distance between the query vector and every stored chunk vector using the HNSW index, returning the top-K nearest neighbours. This lane finds semantically relevant chunks even when they use completely different wording than the question.

cosine similarity HNSW index 3072 dims k=8

🔤 Full-text BM25 Search

In parallel, Azure AI Search runs a classic BM25 keyword search on the raw question text against the text field of every chunk. BM25 scores based on term frequency normalised by document length and corpus-wide inverse document frequency — it favours chunks that contain rare, specific terms from the query. This lane catches exact terminology matches that a vector search might rank lower due to paraphrasing.

BM25 TF-IDF keyword match full-text

∑ RRF Fusion — how @search.score is computed

Azure AI Search merges the two ranked lists using Reciprocal Rank Fusion (RRF). Each document receives a score contribution from each list it appears in:

score(doc) = 1 / (rank_vector + 60) + 1 / (rank_bm25 + 60)

The constant 60 softens the gap between ranks (so rank #1 vs #2 matters less than rank #1 vs #100). A document that ranks #1 in both lists scores approximately 0.0161 + 0.0161 = 0.0323. A document appearing only in one list scores half that. Documents absent from a list contribute 0 from that lane. The final merged list is sorted descending by this fused score — this is the @search.score value captured on each result. The top 15 results proceed to the semantic reranker.

RRF @search.score typical range 0.01–0.06

🧠 Semantic Reranker — @search.reranker_score

After RRF, the top-15 results are re-scored by Azure AI Search's Semantic Reranker — a neural cross-encoder model trained specifically for passage ranking. Unlike RRF (which is purely rank-based), the semantic reranker reads both the question and the full chunk text together and predicts a true relevance score on a 0–4 scale:

0 = completely irrelevant  ·  1 = marginally relevant  ·  2 = somewhat relevant  ·  3 = relevant  ·  4 = highly relevant

The results are re-ordered by @search.reranker_score and this score — not the original RRF score — is used for all downstream confidence decisions. Requires Azure AI Search Standard S1 or higher tier. Controlled by AZURE_SEARCH_SEMANTIC_ENABLED.

Semantic Reranker @search.reranker_score 0–4 scale neural cross-encoder S1+ tier required

🚫 Minimum Score Filter

After reranking, chunks with @search.reranker_score < 1.0 are dropped before the context window is built. This prevents weakly-matched documents from polluting the context sent to GPT-4o. The floor is set by MIN_RERANKER_SCORE = 1.0 in query.py and can be adjusted if queries are too restrictive.

When the semantic reranker is disabled (fallback mode), this filter is not applied and all 15 RRF results pass through to the context builder.

MIN_RERANKER_SCORE = 1.0 pre-context filtering

⬡ Source Type Scoring Profile

An optional tag-based scoring profile (source-boost) can amplify results from a specific source type during the RRF phase. The index stores a source_tags field (["wiki"], ["code"], or ["test"]) for every chunk. When BOOST_SOURCE_TYPE is set in .env, matching documents receive a 3× multiplicative boost applied to their BM25 score before RRF fusion.

Example: BOOST_SOURCE_TYPE=wiki prioritises documentation answers. BOOST_SOURCE_TYPE=test prioritises test case results. Leave empty (default) for equal treatment of all sources.

Part 2 — How confidence is determined

Confidence Signal — decision gate flowchart

Gate 1 — No results

If the hybrid search returned zero chunks (the index is empty, or the query produced no meaningful matches), confidence is immediately set to low. There is no content to base an answer on.

Gate 2 — LLM uncertainty language (checked before scores)

Even when chunks are retrieved and scored, if GPT-4o's answer contains any of the following phrases the response is classified low regardless of score:

"i don't know"  ·  "not enough information"  ·  "cannot find"  ·  "no information"  ·  "not mentioned"  ·  "unclear"

This gate takes priority over the score gates — it reflects the model's own assessment that the retrieved context was insufficient to answer the question.

Gate 3 — Reranker score thresholds

If neither earlier gate fired, confidence is determined by the @search.reranker_score of the top result (0–4 scale from the semantic reranker) combined with the number of chunks that passed the minimum score filter:

Condition Signal Meaning
reranker_score ≥ 2.5  &&  chunks ≥ 3 high Neural cross-encoder rated the top chunk as clearly relevant (≥ 2.5/4), backed by at least 3 corroborating chunks. The answer is well-supported across multiple sources.
reranker_score ≥ 1.5  &&  chunks ≥ 2 medium Moderate semantic relevance — the reranker found a reasonable match and there is at least one corroborating chunk. Answer is likely correct but context coverage is partial.
anything else low Low reranker score or very few chunks survived the min-score filter. The answer may be speculative or based on weakly-matched content.

Fallback (semantic disabled): when AZURE_SEARCH_SEMANTIC_ENABLED=false, the original RRF @search.score thresholds apply instead: > 0.03 + 4 chunks → HIGH; > 0.01 + 2 chunks → MEDIUM; else LOW.
