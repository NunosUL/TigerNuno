🔀 Commit Diff Pipeline

A deep-dive into how TigerChat turns git commit history into searchable, LLM-readable unified diffs — enabling precise answers to questions like the Product Feature Questionnaire (PFQ) that ask "what exactly changed in the code for this User Story?"

End-to-end flow — ingestion + query time

Step by step — ingestion

①

📋 Work Item has ArtifactLink Relations

When a developer associates a commit with a Bug or Task in Azure DevOps (during a PR or manual link), DevOps stores a vstfs:///Git/Commit/… URL in the work item's relations array with rel = "ArtifactLink".

Important: The batch Work Items API (GET /wit/workitems?ids=…&$expand=all) silently omits ArtifactLinks even with $expand=all. TigerChat works around this by fetching each Bug/Task individually — only the single-item endpoint reliably returns ArtifactLinks.

Bugs & Tasks only individual fetch
🔗 raw relation from work item response
{
  "rel":        "ArtifactLink",
  "url":        "vstfs:///Git/Commit/4a9c3e12%2fbd5f7a8c-1234-5678-abcd-ef1234567890%2fa1b2c3d4e5f6...",
  "attributes": { "name": "Fixed in Commit" }
}  // %2f is URL-encoded "/" — must decode before parsing
②

🔗 Decode the vstfs URL

The relation URL is vstfs:///Git/Commit/{projectId}%2f{repoId}%2f{commitSHA}. Two transformations are applied before regex-matching:

  • 1. Strip whitespace and any null bytes (\x00)
  • 2. urllib.parse.unquote() — converts %2f → /
  • 3. Regex match: vstfs:///Git/Commit/(?:[^/]+/)?([^/]+)/([a-fA-F0-9]{40})

The optional (?:[^/]+/)? prefix absorbs the projectId GUID when present, making the pattern work with both the 3-segment and legacy 2-segment URL formats.

🔎 decoded URL and extracted components
// After URL-decode:
vstfs:///Git/Commit/4a9c3e12/bd5f7a8c-1234-5678-abcd-ef1234567890/a1b2c3d4e5f6...

// Regex groups:
group(1) = "bd5f7a8c-1234-5678-abcd-ef1234567890"  // repoId (GUID)
group(2) = "a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2"  // commitSHA (40 hex chars)

Similarly, branch links use vstfs:///Git/Ref/…/{hexEncodedRefPath} — the hex string is decoded as UTF-8 bytes to recover the branch name (e.g. refs/heads/feature/marketplace).

③

📡 Fetch Commit Metadata & Parent SHA

Two concurrent API calls retrieve everything needed to build the diff:

commit details
GET /git/repositories/{repoId}
    /commits/{commitSHA}

"comment": "Upgrade to .NET 10"
"author":  { "name": "Dev Name",
              "date": "2026-04-20" }
"parents": ["f0e9d8c7b6a5..."]
// parents is a list of plain strings
// (Azure DevOps API v7.1)
// NOT a list of dicts
changed files
GET /git/repositories/{repoId}
    /commits/{commitSHA}/changes

"changes": [
  { "item": { "path": "/WERCSmart.DataAccess/UL.Wercsmart.DataAccess.csproj",
              "isFolder": false },
    "changeType": "edit" },
  // ... one entry per changed file
]
⚠️
API format pitfall — the parents array contains plain SHA strings (e.g. ["f0e9d8c7b6a5…"]), not objects. Calling parents[0].get("objectId") throws 'str' object has no attribute 'get' — a silent crash that caused all commit diffs to be silently dropped on every run. TigerChat normalises with a _parent_sha() helper: if the element is a dict, use .get("objectId"); otherwise cast to string directly.
GET /commits/{sha} GET /commits/{sha}/changes
④

📂 Fetch File Content at Both SHAs

For each text file in the changed-files list (skipping non-text extensions and files over 30 KB), two API calls fetch the file content as plain text using the $format=text parameter and versionType=commit:

file AFTER change (at commitSHA)
GET /git/repositories/{repoId}/items
  ?path=/WERCSmart.DataAccess/
         UL.Wercsmart.DataAccess.csproj
  &version=a1b2c3d4…  ← new commit
  &versionType=commit
  &$format=text

// returns raw .csproj XML as text
file BEFORE change (at parentSHA)
GET /git/repositories/{repoId}/items
  ?path=/WERCSmart.DataAccess/
         UL.Wercsmart.DataAccess.csproj
  &version=f0e9d8c7…  ← parent commit
  &versionType=commit
  &$format=text

// same file, one commit earlier
💡
Both file versions are addressed by immutable git object SHAs, not by branch names. This means the diff produced today is byte-for-byte identical to the diff produced in 6 months — regardless of any subsequent commits or branch changes.
both adressed by SHA max 30 KB per file
⑤

⚗️ Generate Unified Diff with Python difflib

The diff is generated locally in Python using the standard library's difflib.unified_diff() — no external diff service or API call. Context lines are set to 4 (n=4) to give the LLM enough surrounding code to understand the change. Output is capped at 200 lines per file.

+++ UL.Wercsmart.DataAccess.csproj (after)

-    <TargetFramework>net8.0</TargetFramework>
+    <TargetFramework>net10.0</TargetFramework>
     <Nullable>enable</Nullable>
     <ImplicitUsings>enable</ImplicitUsings>
   </PropertyGroup>
@@ -15,8 +15,8 @@
   <ItemGroup>
-    <PackageReference Include="Microsoft.Extensions.Hosting" Version="9.0.9" />
+    <PackageReference Include="Microsoft.Extensions.Hosting" Version="10.0.0" />
-    <PackageReference Include="Microsoft.Extensions.DependencyInjection" Version="9.0.9" />
+    <PackageReference Include="Microsoft.Extensions.DependencyInjection" Version="10.0.0" />
   </ItemGroup>

For .csproj files specifically, the ingester also parses both versions as XML and extracts a structured NuGet package diff table — a much more readable summary for the LLM than raw diff lines:

⑥

💾 Build & Store the Diff Document

All changed files' diffs are assembled into a single Markdown document and stored in Azure AI Search at path /commit-diffs/{40-char-commitSHA}. Because the path is keyed on the immutable SHA, the document is deduplicated — if two work items share the same commit, only one diff document is ever created.

The Markdown document also embeds *sha:{40-char-sha}* markers, which are used at query time to retrieve this exact document.

💾 assembled diff document markdown (abbreviated)
# Commit a1b2c3d
*sha:a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2*
**Message:** Upgrade packages to .NET 10
**Author:** Dev Name  |  **Date:** 2026-04-20
**Compared against parent:** f0e9d8c
**Linked Work Item:** 301827

## Changed Files

### `/WERCSmart.DataAccess/UL.Wercsmart.DataAccess.csproj` — edit

**NuGet package changes in `UL.Wercsmart.DataAccess.csproj`:**

| Package | Before | After |
|---------|--------|-------|
| TargetFramework | `net8.0` | `net10.0` |

At query time — how the diff reaches the LLM

1
User asks a PFQ question

e.g. "Review PFQ for User Story 301701 — what changed in authentication, APIs, or packages?"

2
Hybrid search retrieves Work Item chunks

The query vector matches against the /work-items/301701 Markdown chunks, which contain the work item description, linked children/parents, and the ## Git Commits section listing each commit with *sha:40-char-hex* markers.

3
SHA markers are extracted from the retrieved text

query.py runs a regex (r"\*sha:([a-f0-9]{40})\*") over every retrieved chunk. Each unique SHA found triggers a look-up of the corresponding /commit-diffs/{sha} path in the search index.

4
Diff chunks are retrieved and injected into context

The diff documents are filtered from the search index by path prefix, retrieved, and their chunks are appended to the context window that is sent to GPT-4o. Each diff chunk is labelled [Commit Diff] in the context to help the LLM attribute it correctly.

5
GPT-4o answers with real code-level detail

Armed with the actual unified diff, NuGet table, file paths, and package versions, the LLM can answer PFQ sections accurately — reporting "TargetFramework changed from net8.0 to net10.0 in WERCSmart.DataAccess.csproj" instead of "Not applicable".

💡

Why this design works

The diff is generated once at ingestion time and stored as a first-class search document. This means the LLM doesn't need code-execution tools or live repo access — it just reads the pre-computed, human-readable diff from the search index like any other document. The *sha:…* marker in the Work Item document acts as a typed foreign key linking the work item to its commit diff, enabling the query pipeline to automatically join them at retrieval time.

The immutability of git SHAs means the diff is always the same regardless of when ingestion runs — there is no "stale diff" problem even if a repository is rebased or branches are deleted after the fact.
