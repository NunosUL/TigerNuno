How It Works

A fully automated RAG ingestion pipeline that crawls five Azure DevOps sources — Wiki pages, Git repository code files, Test Management cases, Work Items (Epics, Features, User Stories, Tasks & Bugs), and linked Commit Diffs — then chunks, embeds, and indexes all content into Azure AI Search for retrieval-augmented generation.

Blob Storage holds two things: a wiki_ORG_PROJECT.jsonl snapshot overwritten on each crawl, and a wiki_hashes.json manifest that tracks which records have changed to avoid re-embedding unchanged content. Path prefixes guarantee no collisions across sources: /wiki/… · /repos/… · /test-cases/… · /work-items/… · /commit-diffs/…

Pipeline Steps

1

Crawl

Connects to Azure DevOps using a PAT token and collects records from up to five sources — each controlled by an environment toggle and a UI selector:

  • 📄 Wiki pages — lists all pages recursively across all project and code wikis, fetches each page's Markdown plus its inline discussion comments, converts to HTML, and extracts links.
  • 💻 Code files — enumerates every Git repository, walks the file tree (paginated via x-ms-continuationtoken), downloads files up to 100 KB, skips binaries, lock files, migrations, and auto-generated code. Supports selective crawl by repository name.
  • 🧪 Test cases — traverses all Test Plans and Suites, deduplicates work items by ID across plans, resolves Shared Step references inline, and formats each test case as structured Markdown with a steps table, plan/suite context, discussion comments, and tags.
  • 📋 Work Items — fetches Epics, Features, User Stories, Tasks, and Bugs via per-type WIQL queries (one per type to stay under the 20,000-item limit). Each record includes description, acceptance criteria, repro steps, linked work items (parent/children/related), discussion comments, code attachments, and area-path / iteration metadata. Supports selective crawl by area path tree.
  • 🔀 Commit Diffs — automatically triggered for Bugs and Tasks that have linked git commits (via ArtifactLink relations). For each linked commit SHA, fetches the changed files at both commit and parent SHA, then generates a unified diff using Python's difflib. NuGet package changes in .csproj files are extracted into a before/after table. Stored at /commit-diffs/{sha} — one document per unique commit SHA, deduplicated across all work items.
    → Deep dive: Commit Diff Pipeline

All records share the same {path, remote_url, html, markdown, links, crawled_at} shape and are saved as a single JSONL blob. Discussion threads are fetched in parallel (8 workers). Any item that fails is logged and skipped without stopping the run.

2

Chunk

Each record's Markdown is split into overlapping chunks using LangChain's RecursiveCharacterTextSplitter. The splitter respects heading boundaries (## , ### ) before falling back to paragraph and sentence breaks. Chunk size is 1,000 characters with a 150-character overlap to preserve context at boundaries.

For test cases, a short identity prefix (Test Case {id} · {title}) is prepended to every non-first chunk so each chunk remains discoverable by TC ID or title even after splitting. Chunking runs across 8 parallel workers for speed. Only records whose MD5 hash has changed are chunked.

3

Embed

Each chunk's text is sent to the Azure OpenAI embeddings API in batches of 200 chunks using 3 parallel workers with a 0.5 s inter-batch delay to stay within the Azure OpenAI S0 quota (~60 RPM). text-embedding-3-large produces a 3,072-dimensional vector capturing semantic meaning. Rate-limit (429) errors trigger automatic retry with exponential back-off honouring the API's Retry-After hint.

Tuning: set EMBED_INTER_BATCH_DELAY_S=1.5 in .env to slow down if you hit 429s; lower to 0.1 if your Azure quota is higher.

4

Index

Ensures the wiki-index index exists in Azure AI Search. The schema includes source_type (filterable/facetable string — values: wiki, code, test, workitem, commit-diff) and source_tags (collection field used by the source-boost scoring profile to amplify results from a specific source by 3×). Vector search uses HNSW with cosine metric. A semantic configuration (tigerchat-semantic) enables the semantic reranker for hybrid search.

Azure AI Search HNSW · cosine semantic reranker source-boost profile

5

Upload

Upserts all chunks into Azure AI Search in batches of 200 documents using 4 parallel threads. Because the operation is an upsert keyed on chunk id, re-running the pipeline never creates duplicates — changed records overwrite in place, new records are created. After a successful upload with zero failures, the hash manifest is updated in blob storage. If any failures occur, the manifest is intentionally not saved so the next run retries affected documents.

Azure AI Search upsert · 4 workers · batch 200

Ingestion UI — Pipeline Settings

Crawl Toggle, Source Selection & Selective Pickers

The ingestion page exposes granular controls so you can target exactly what runs without committing to a full crawl-to-upload cycle every time:

⚙️
Include crawling — master toggle. When ON (default), fresh content is pulled from Azure DevOps before indexing. When OFF, the pipeline loads the last saved JSONL snapshot from Blob Storage and re-processes it — useful after index schema changes or model upgrades without spending API quota.
☑️
Sources — four toggles: Wiki, Source Code, Test Plans, Work Items. Deselect any to skip it — e.g. crawl only Work Items after a sprint, without re-crawling all repositories.
📦
Repository picker — when Source Code is selected, a searchable list of all Git repositories lets you select a subset. Leave all unchecked to crawl every non-empty, non-disabled repo.
🧪
Test Plan cascade — drill down from Plan → Suite → individual Test Cases to crawl exactly the test scope you need.
🗂️
Work Items area picker — an expandable area-path tree lets you scope the Work Items crawl to specific teams or product areas, keeping the index focused and avoiding unrelated backlogs.
💡
All selections are passed as parameters to GET /ingest/stream and override the CRAWL_WIKI / CRAWL_CODE / CRAWL_TESTS / CRAWL_WORK_ITEMS environment variables for that run only.

Change Detection

Hash Manifest

To avoid re-embedding unchanged content (and wasting Azure OpenAI tokens), the pipeline maintains a wiki_hashes.json manifest in Blob Storage. It maps the path of every crawled record to an MD5 hash of its Markdown content. All five source types share the same manifest because their paths never collide.

📥
Load wiki_hashes.json from blob storage (empty dict on first run)
🔍
Hash each crawled record's Markdown with MD5 and compare to the stored hash
✓
Unchanged records — skipped entirely (no chunking, no embedding, no upload)
⚡
New or changed records — chunked, embedded, and upserted into the index
💾
Manifest saved back to blob only if upload completes with zero failures
🔒 wiki_hashes.json · all five source types in one manifest
{
  "/wiki/WERCSmart.Wiki/Home":                                "a3f2c1d8e4b5f6a7",
  "/repos/WERCSmart/src/Services/MarketplaceService.cs":    "c2a8f1b4d7e3c9a1",
  "/test-cases/296199":                                      "e1f3b7a2d8c5e9f4",
  "/work-items/301701":                                      "b7e3a2d1c9f4e8b2",  // changed → re-embed
  "/commit-diffs/a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2":  "d4b9f2e1a7c3d8f6",

// records whose MD5 matches → skipped entirely, zero API calls
