How It Works

A fully automated RAG ingestion pipeline that crawls an Azure DevOps Wiki, chunks and embeds the content, and indexes it into Azure AI Search for retrieval-augmented generation.

Pipeline Steps

1

Fetch

Scans the Azure Blob Storage container for .jsonl files and picks the most recently modified one. Each file is a full snapshot of the wiki produced by crawler.py, with one JSON record per page containing its path, URL, raw Markdown, rendered HTML, and crawl timestamp.

Azure Blob Storage
📄 page record · one entry in the .jsonl blob
{
  "path": "/Documentation/Setup",
  "remote_url": "https://dev.azure.com/org/project/_wiki/wikis/project.wiki/5/Setup",
  "markdown": "# Setup Guide\n\nTo install the project, run `npm install`\nfrom the root directory. Ensure Node.js 18+...",
  "html": "<h1>Setup Guide</h1><p>To install the project, run ...</p>",
  "links": ["/Documentation/Prerequisites", "/Guides/Deploy"],
  "crawled_at": "2026-04-08T00:24:38Z"
}
2

Chunk

Each page's Markdown is split into overlapping chunks using LangChain's RecursiveCharacterTextSplitter. The splitter respects heading boundaries (##, ###) before falling back to paragraph and sentence breaks. Chunk size is 1000 characters with 150-character overlap to preserve context at boundaries. Only pages whose content has changed since the last run are chunked (see change detection below).

LangChain
✂️ chunk record · after splitting · before embedding
{
  "id": "documentation_setup__0",  // "{safe_path}__{chunk_index}"
  "text": "# Setup Guide\n\nTo install the project, run `npm install` from\nthe root directory. Ensure Node.js 18+ is installed before\nrunning the command.\n\n## Prerequisites\n\nThe following tools must be installed on your machine...",
  "path": "/Documentation/Setup",
  "url": "https://dev.azure.com/org/project/_wiki/wikis/project.wiki/5/Setup",
  "crawled_at": "2026-04-08T00:24:38Z"
  // ~1000 chars of text  ·  no embedding yet — added in step 3
}
3

Embed

Each chunk's text is sent to the Azure OpenAI embeddings API in batches of 100. The text-embedding-3-large model produces a 3072-dimensional vector that captures the semantic meaning of the text. These vectors are what enable similarity search at query time.

Azure OpenAI text-embedding-3-large
⚡ embedded chunk · 3072-dimensional vector added
{
  "id": "documentation_setup__0",
  "text": "# Setup Guide\n\nTo install the project, run `npm install`...",
  "path": "/Documentation/Setup",
  "url": "https://dev.azure.com/org/project/_wiki/wikis/project.wiki/5/Setup",
  "crawled_at": "2026-04-08T00:24:38Z",
  "embedding": [
    0.0234, -0.0891, 0.1102, -0.0338, 0.0567, -0.1245, 0.0789, 0.0423,
    ... 3,064 more float32 values  (text-embedding-3-large = 3072 dims total) ...
  ]
}
4

Index

Ensures the wiki-index index exists in Azure AI Search. If not, it creates it with fields for id, text, path, url, crawled_at, and the embedding vector field configured with HNSW (Hierarchical Navigable Small World) — a fast approximate nearest-neighbour algorithm optimised for high-dimensional vectors.

Azure AI Search HNSW
🗂️ index schema · wiki-index field definitions
{
  "name": "wiki-index",
  "fields": [
    { "name": "id",         "type": "Edm.String",            "key": true,  "filterable": true  },
    { "name": "text",       "type": "Edm.String",            "searchable": true                    },
    { "name": "path",       "type": "Edm.String",            "filterable": true, "facetable": true },
    { "name": "url",        "type": "Edm.String"                                           },
    { "name": "crawled_at", "type": "Edm.String",            "filterable": true                    },
    { "name": "embedding",  "type": "Collection(Edm.Single)",
      "dimensions": 3072,  "vectorSearchProfile": "hnsw-profile"                  }
  ],
  "vectorSearch": {
    "algorithms": [{ "name": "hnsw-config", "kind": "hnsw",
      "hnswParameters": { "metric": "cosine", "m": 4 } }]
  }
}
5

Upload

Upserts all chunks (text + metadata + embedding vector) into Azure AI Search in batches of 100. Because the operation is an upsert keyed on id, re-running the pipeline never creates duplicates — changed pages are overwritten in place. After a successful upload with zero failures, the hash manifest is updated in blob storage.

Azure AI Search upsert
📤 upsert response · per batch of 100 chunks
{
  "value": [
    { "key": "documentation_setup__0", "status": true, "statusCode": 201 },  // new doc created
    { "key": "documentation_setup__1", "status": true, "statusCode": 201 },  // new doc created
    { "key": "documentation_setup__2", "status": true, "statusCode": 200 },  // existing doc overwritten
    ... up to 100 results per batch call ...
  ]
}
// 201 = created  ·  200 = updated (upsert)
// hash manifest only saved after all batches return zero failures

Change Detection

To avoid re-embedding unchanged content (and wasting Azure OpenAI tokens), the pipeline maintains a wiki_hashes.json manifest in blob storage. It maps each wiki page path to an MD5 hash of its Markdown content.

📥
Load wiki_hashes.json from blob storage (empty on first run)
🔍
Hash each page's Markdown and compare to the stored hash
✓
Unchanged pages — skipped entirely (no chunking, no embedding, no upload)
⚡
New or changed pages — chunked, embedded, and upserted
💾
Manifest saved back to blob only if upload succeeds with zero failures
🔒 wiki_hashes.json · stored in blob storage
{
  "/Home":                         "a3f2c1d8e4b5f6a7",
  "/Documentation/Setup":         "b7e3a2d1c9f4e8b2",  // changed → will re-embed
  "/Documentation/API":           "c2a8f1b4d7e3c9a1",
  "/Guides/Onboarding":           "d9b2e7a3f1c8d4b6",
  "/Guides/Deploy":               "e1f3b7a2d8c5e9f4",
  ... one MD5 hash per wiki page (296 total in this example) ...
}
// pages whose current MD5 != stored hash are chunked + embedded + upserted
// pages whose MD5 matches are skipped entirely — zero API calls

Crawler (crawler.py)

Runs separately from the ingestion pipeline. It authenticates to Azure DevOps using a Personal Access Token (PAT) and calls the Wiki REST API to list all pages recursively in a single request, then fetches each page's Markdown content. The Markdown is also converted to HTML server-side for flexibility.

Each run produces a new timestamped .jsonl blob (e.g. wiki_ulpsi_NetProjects10_20260408T002438Z.jsonl). The ingestion pipeline always picks the most recent one automatically.
