How It Works

A fully automated RAG ingestion pipeline that crawls three Azure DevOps sources — Wiki pages, Git repository code files, and Test Management cases — then chunks, embeds, and indexes all content into Azure AI Search for retrieval-augmented generation.

Blob Storage is used for two things: a fixed-name JSONL snapshot (wiki_ORG_PROJECT.jsonl) is overwritten on each crawl run, and a hash manifest (wiki_hashes.json) tracks which records have changed between runs to avoid re-embedding unchanged content.

Pipeline Steps

1

Crawl

Connects to Azure DevOps using a PAT token and collects records from up to three sources — each controlled by an environment toggle:

  • Wiki pages — lists all pages recursively (recursionLevel=full), fetches each page's Markdown, converts to HTML, and extracts links.
  • Code files — enumerates every Git repository, walks the file tree (paginated via x-ms-continuationtoken), downloads files up to 100 KB, skips binaries, lock files, migrations, and auto-generated code.
  • Test cases — traverses all Test Plans and Suites, deduplicates work items, and formats each test case as structured Markdown with a steps table, plan/suite context, and tags — enabling queries like "what test cases cover Onboarding?"

All records share the same {path, remote_url, html, markdown, links, crawled_at} shape and are written as a single .jsonl blob in Azure Blob Storage (always the same filename — no timestamp suffix — so each run overwrites the previous snapshot). Any item that fails to fetch is logged and skipped without stopping the run.

2

Chunk

Each page's Markdown is split into overlapping chunks using LangChain's RecursiveCharacterTextSplitter. The splitter respects heading boundaries (##, ###) before falling back to paragraph and sentence breaks. Chunk size is 1000 characters with 150-character overlap to preserve context at boundaries. Only pages whose content has changed since the last run are chunked (see change detection below).

3

Embed

Each chunk's text is sent to the Azure OpenAI embeddings API in batches of 100. The text-embedding-3-large model produces a 3072-dimensional vector that captures the semantic meaning of the text. These vectors are what enable similarity search at query time.

4

Index

Ensures the wiki-index index exists in Azure AI Search. If not, it creates it with fields for id, text, path, url, crawled_at, and the embedding vector field configured with HNSW (Hierarchical Navigable Small World) — a fast approximate nearest-neighbour algorithm optimised for high-dimensional vectors.

Azure AI Search HNSW

  ],
  "vectorSearch": {
    "algorithms": [{ "name": "hnsw-config", "kind": "hnsw",
      "hnswParameters": { "metric": "cosine", "m": 4 } }]
  }
}
5

Upload

Upserts all chunks (text + metadata + embedding vector) into Azure AI Search in batches of 100. Because the operation is an upsert keyed on id, re-running the pipeline never creates duplicates — changed pages are overwritten in place. After a successful upload with zero failures, the hash manifest is updated in blob storage.

Azure AI Search upsert
📤 upsert response · per batch of 100 chunks
{
  "value": [
    { "key": "documentation_setup__0", "status": true, "statusCode": 201 },  // new doc created
    { "key": "documentation_setup__1", "status": true, "statusCode": 201 },  // new doc created
    { "key": "documentation_setup__2", "status": true, "statusCode": 200 },  // existing doc overwritten
    ... up to 100 results per batch call ...
  ]
}
// 201 = created  ·  200 = updated (upsert)
// hash manifest only saved after all batches return zero failures

Ingestion UI — Pipeline Settings

Crawl Toggle & Source Selection

The ingestion page exposes two settings that let you control exactly what runs before committing to a full crawl-to-upload cycle:

⚙️
Include crawling — master toggle. When ON (default), fresh content is pulled from Azure DevOps before indexing. When OFF, the pipeline loads the last saved JSONL snapshot from Blob Storage and re-indexes it without making any DevOps API calls. Useful after index schema changes or embedding model upgrades.
☑️
Sources to crawl — visible only when crawling is ON. Three toggles: Wiki, Source Code, Test Plans. Deselect a source to skip it in this run — e.g. crawl only Test Plans after adding new test cases, without re-crawling all 90+ repositories.
💡
Settings are passed as query parameters to GET /ingest/stream and override the CRAWL_WIKI / CRAWL_CODE / CRAWL_TESTS environment variables for that run only.

Change Detection

Hash Manifest

To avoid re-embedding unchanged content (and wasting Azure OpenAI tokens), the pipeline maintains a wiki_hashes.json manifest in Blob Storage. It maps the path of every crawled record — wiki page, code file, or test case — to an MD5 hash of its Markdown content. All three source types share the same manifest because their paths never collide (/Documentation/…, /repos/…, /test-cases/…).

📥
Load wiki_hashes.json from blob storage (empty on first run)
🔍
Hash each record's Markdown and compare to the stored hash
✓
Unchanged records — skipped entirely (no chunking, no embedding, no upload)
⚡
New or changed records — chunked, embedded, and upserted
💾
Manifest saved back to blob only if upload succeeds with zero failures
🔒 wiki_hashes.json · stored in blob storage · all source types in one manifest
{
  "/Home":                                          "a3f2c1d8e4b5f6a7",
  "/Documentation/Setup":                          "b7e3a2d1c9f4e8b2",  // changed → will re-embed
  "/repos/MyApp/src/Services/PaymentService.cs":   "c2a8f1b4d7e3c9a1",
  "/repos/MyApp/src/Controllers/HomeController.cs": "d4b9f2e1a7c3d8f6",  // changed → will re-embed
  "/test-cases/12345":                              "e1f3b7a2d8c5e9f4",
  "/test-cases/12346":                              "f5a2c9e3b1d7f4a8",
  ... one MD5 per record across wiki, code, and test sources ...
}
// records whose current MD5 != stored hash are chunked + embedded + upserted
// records whose MD5 matches are skipped entirely — zero API calls
