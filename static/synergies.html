Pipeline Synergies

The ingestion pipeline and the RAG chat are two separate processes — but every decision made in ingestion was built to make chat work correctly. Five sources, one shared index, and a commit diff pipeline that closes the loop between backlog and code reality.

⚙ Ingestion 🔀 Commit Diffs ⬡ Shared Contract 💬 Chat / RAG
📄 Wiki Pages 💻 Code Files 🧪 Test Cases 📋 Work Items 🔀 Commit Diffs

The 9 synergies in detail

⚡
① Same Embedding Model

Both sides use text-embedding-3-large. This is the most critical constraint — vectors only make sense for similarity search if they were created in the same 3,072-dimensional space. Different models = meaningless results, silently.

text-embedding-3-large 3072 dims critical constraint
⬡
② Shared Index — the Handoff Point

Azure AI Search's wiki-index is where ingestion ends and chat begins. Ingestion writes; chat reads. The index schema — including source_type, source_tags, and the scoring profile — was designed to serve both pipelines.

wiki-index upsert by id hybrid search
✂️
③ Chunk Size Shapes Answer Quality

The 1,000-char / 150-overlap split was chosen so 8–10 chunks fit in GPT-4o's 16,000-char context budget while remaining semantically complete. Commit diff chunks are often larger but capped to keep the budget balanced.

1000 chars 150 overlap 16k budget
🔑
④ Deterministic IDs Enable Clean Upserts

Chunk IDs are built deterministically from path + chunk index. Re-running ingestion overwrites exactly the right vectors — never creating duplicates. Commit diff IDs use the 40-char SHA, guaranteeing uniqueness across all work items.

path-based ID SHA-based diff ID no duplicates
📄
⑤ Metadata Becomes Citations

path and url stored at ingestion are retrieved alongside every chunk at query time and surfaced as clickable source citations. Commit diff chunks cite back to the Azure DevOps commit page via remote_url.

path url source citations
🔒
⑥ Change Detection Keeps Index Fresh

The MD5 hash manifest means only changed records get re-embedded. Commit diff records are immutable (same SHA = same diff) so once ingested they are never re-processed. If upload fails, the manifest is not saved — next run retries safely.

MD5 per record skip unchanged SHA immutability
🔍
⑦ Text Field Enables Hybrid Search

The text field stored at ingestion is marked searchable, enabling BM25 full-text search alongside vector search. Commit diff chunks contain real keywords (file names, package names, function signatures) that BM25 excels at matching.

BM25 keywords vector similarity RRF fusion
📋
⑧ Work Items as Structured Context

Work Items add a dimension that Wiki and Code can't provide: intent. A User Story provides acceptance criteria; a Bug has repro steps; an Epic defines the business goal. The LLM uses this to answer "why was this change made?" — not just "what changed".

acceptance criteria area path linked WIs
🔀
⑨ Commit Diffs Close the Code Evidence Loop

Work Item chunks contain *sha:40-char-hex* markers. At query time, query.py scans retrieved chunks for these markers and fetches the corresponding /commit-diffs/{sha} documents, injecting actual before/after file diffs into the context. This is what enables questions related to filling a PFQ document to return real answers like "TargetFramework: net8.0 → net10.0".

sha markers diff lookup PFQ precision

The key insight

The ingestion pipeline and RAG chat are decoupled in time — ingestion runs on demand, chat runs on every question — but tightly coupled in contract: same embedding model, same index schema, same metadata fields. Change any one and both sides break.

The commit diff pipeline adds a second contract: the *sha:…* markers in Work Item Markdown are a typed foreign key into /commit-diffs/{sha}. This makes the query pipeline a two-hop retrieval system — Work Item chunk → SHA marker → diff chunk — without any explicit join logic in the LLM. The LLM simply receives a complete context bundle and answers as if it had read the code change itself.
